## Summary

When reopening a saved anonymous deal, clicking Next showed "Please complete required fields" then worked after triggering blur validation on zipCode. Multiple bugs in the duplicate/reopen deal flow were identified and fixed across frontend and backend.

## Bugs Found and Fixed

---

### MMH-1357-BUG-001: party.zipCode never assigned after zipCode search

- **Title**: `party.zipCode` never assigned after zipCode API search
- **Root Cause**: In `applyZipCodeToPartyFields`, after a successful zipCode API search (e.g., "28001" returns `{ zipCode: "28001", city: "Madrid", country: "ES" }`), the function only updated `party.city` and `party.country` — but NEVER assigned `party.zipCode`. When `applyZipCodeResultToParty` called `validateField({ value: party?.zipCode ?? "" })`, the value was empty, causing a "required" validation error.
- **Files**:
  - `hyperfront/features/hyperFront/stores/partiesModule.ts`
- **Fix**: Added `party.zipCode = zipCode?.zipCode ?? party.zipCode ?? "";` in `applyZipCodeToPartyFields` before city/country assignment.
- **Impact**: Next button was blocked by "Please complete required fields" error for deals with validated zipCode.

---

### MMH-1357-BUG-002: offerStore used without declaration in validateCommunicationField

- **Title**: `offerStore` used without declaration in `validateCommunicationField`
- **Root Cause**: Function `validateCommunicationField` used `offerStore.workflow` but `offerStore` was not declared. This caused a `ReferenceError` blocking deal step navigation.
- **Files**:
  - `hyperfront/shared/utils/index.ts`
- **Fix**: Added `const offerStore = useOfferStore();` at the beginning of the function.
- **Impact**: Deal step navigation was blocked with `ReferenceError`.

---

## Related Fixes (deployed alongside)

### Fix A: ignore_warnings on POST /sales-quotes/{id}/customers
- **Component**: `deal-bs` backend — `DealServiceImpl.java`
- **Bug**: 422 UNPROCESSABLE_ENTITY when duplicating without customer data
- **Fix**: Added `?ignore_warnings=true` query param to the API call

### Fix B: Duplicate response used wrong customerId
- **Component**: `hyperfront/server/api/deals/duplications.post.tsx` (lines 167-169, 187)
- **Bug**: Override `duplicateResponse.quote.customer.customerId` with the anonymous customerId to prevent mismatched references

### Fix C: Merge logic to preserve user-entered data
- **Component**: `hyperfront/server/api/parties/[customerId]/infos.get.ts` (lines 142-143, 158, 184)
- **Bug**: Miles API returns empty fields, overwriting user-entered data on party info fetch
- **Fix**: Added merge logic — preserve existing data for fields Miles returns empty

### Fix D: Inject customerId into party store
- **Component**: `hyperfront/composables/useInit.ts` (line 575)
- **Bug**: MongoDB minimal party caused "undefined undefined" client name display
- **Fix**: `initMinimalClientParty` now injects `customerId` from deal root `dealData.customer_id`

### Fix E: PayloadBuilder return null instead of objectId hash
- **Component**: `deal-bs` backend — `PayloadBuilder.java`
- **Bug**: Empty fiscalCode generated hash `6a01a9fdd5106ed0746ad34c` instead of null
- **Fix**: Return `null` instead of `objectId.toHexString()` when fiscalCode is empty
