# MMH-1496: Customer Data Missing in Deal List After Save (Visible in Miles)

## Summary

After saving a deal, customer data (entity type, client type) showed as empty/null in the hyperfront deal list, even though the data was correctly visible in Miles. Root cause was Miles backend disabling `enumGroupId` return, which broke the frontend's entity group resolution.

## Root Cause

Miles backend stopped returning `enumGroupId` in API responses. The `resolveEntityGroupId()` function in `partyMapper.ts` returned null because it relied solely on `enumGroupId`/`enumId` from the source object. Additionally, **4 bypass paths** read `customerObj?.legalEntity?.enumGroupId` directly instead of using the resolved value, so all 5 code paths were broken.

**Depth**: Level 2 (Vue composable → deal payload builder → party mapper)

## Fix Applied

### File: `server/utils/partyMapper.ts` (6 changes)

| # | Change | Location | Description |

| 5 | Fixed `buildProposalParty` call | ~line 177 | Pass `legalEntityGroupId` to `buildProposalOrg` |
| 6 | Fixed `mapPartyDetailsSimulation` | ~line 222 | Use `resolveEntityGroupId(source)` instead of raw `source?.legalEntity?.enumGroupId` |

### File: `server/services/dealPayloadBuilder.ts` (1 change)

| # | Change | Location | Description |
|---|--------|----------|-------------|
| 7 | Guard null enumGroupId | ~line 14 | Ternary before `getLabelClientType()` to handle null `entityGroupIdResult` |

### Entity Inference Logic (Change #1)

When `enumGroupId` and `enumId` are both null, the fallback infers entity type from data fields:

- **Individual person (`PERSONNE_PHYSIQUE`)**: `source.individualPerson` is present
- **Sole trader (`PERSONNE_PHYSIQUE`)**: `source.tradingName` is present (without vatNumber)
- **Organization (`PERSONNE_MORALE`)**: `source.vatNumber` is present (typically with tradingName)

## Key Technical Detail
The fix uses **data-driven inference** as fallback: the presence of `individualPerson` vs `tradingName` vs `vatNumber` reliably indicates the entity type (Individual, Sole Trader, Organization). This makes the fix robust even if the Miles API changes its response structure again.
