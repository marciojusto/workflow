# MMH-1373: Modify Button Accessibility for Canceled Deals

## Summary
Fixed bug where the "Modify Deal" button was incorrectly visible for deals with terminal statuses (canceled, rejected, expired). Users could click the modify button even when they shouldn't be able to modify these deals.

## Problem Description
The "Modify Deal" button was accessible for deals in terminal states:
- Canceled deals (both "Canceled" and "Cancelled" spellings)
- Rejected deals
- Expired deals

This allowed users to attempt modifications on deals that should be read-only.

## Root Cause Analysis

### Issue 1: Case-Sensitive Status Comparison
**File**: `composables/useDealStatus.ts`

The original implementation used exact string comparison:
```typescript
isCancelled: computed(() => status.value === CANCELLED_STATUS)
isRejected: computed(() => status.value === REJECTED_STATUS)
isExpired: computed(() => status.value === EXPIRED_STATUS)
```

**Problems identified**:
1. `isCancelled` only matched "Cancelled" (British spelling) but not "Canceled" (American spelling)
2. All comparisons were case-sensitive, failing if backend returned different casing
3. `showModifyButton` in components didn't check for expired status

### Issue 2: Missing Expiration Check
**Files**: `features/deal/Offer.vue`, `features/financing/components/FinancingActions.vue`

The modify button logic only checked for canceled/rejected statuses but not expired status.

## Fix Applied

### 1. Normalized Status Comparisons
**File**: `composables/useDealStatus.ts` (lines 25-36)

Changed all status comparisons to be case-insensitive and handle spelling variations:

```typescript
isRejected: computed(() => 
  status.value?.toLowerCase() === REJECTED_STATUS.toLowerCase()
),
isCancelled: computed(() => {
  const normalizedStatus = status.value?.toLowerCase()
  return normalizedStatus === CANCELLED_STATUS.toLowerCase() || 
         normalizedStatus === 'canceled'.toLowerCase()
}),
isExpired: computed(() => 
  status.value?.toLowerCase() === EXPIRED_STATUS.toLowerCase()
)
```

### 2. Added Expiration Check to Modify Button Logic
**File**: `features/deal/Offer.vue` (line 402, lines 403-409)

```typescript
const { isCancelled, isRejected, isExpired } = useDealStatus()

const showModifyButton = computed(() => {
  return (
    isDeal.value &&
    !isCancelled.value &&
    !isRejected.value &&
    !isExpired.value &&  // Added
    dealStatus.value !== 'DRAFT' &&
    dealStatus.value !== 'PENDING_DOCUMENTS' &&
    dealStatus.value !== 'PENDING_APPROVAL'
  )
})
```

**File**: `features/financing/components/FinancingActions.vue` (lines 118-124)

```typescript
const { isCancelled, isRejected, isExpired } = useDealStatus()

const shouldShowModifyBtn = computed(() => {
  return (
    !isCancelled.value &&
    !isRejected.value &&
    !isExpired.value &&  // Added
    dealStatus.value !== 'DRAFT' &&
    dealStatus.value !== 'PENDING_DOCUMENTS' &&
    dealStatus.value !== 'PENDING_APPROVAL'
  )
})
```

## Technical Considerations

### Spelling Normalization
The fix handles both British ("Cancelled") and American ("Canceled") spellings by:
1. Converting status to lowercase for comparison
2. Explicitly checking for both spelling variants

### Case-Insensitive Matching
All status comparisons now use `.toLowerCase()` to ensure consistent behavior regardless of how the backend returns status strings.

### Terminal State Definition
Terminal deal states that should disable modification:
- Canceled/Cancelled
- Rejected
- Expired
- DRAFT
- PENDING_DOCUMENTS
- PENDING_APPROVAL

## Data Source: Expired Status Origin

The "Expired" status (`hyper_front_status: "Expired"`) is stored directly in MongoDB in the `deal` collection. It is set by two backend mechanisms:

### A) Scheduled Job (Daily)
- **File**: `server/plugins/scheduler.ts` → calls `expireDealsByPhaseDates()` daily
- **File**: `server/services/dealService.ts:593-647`
- **Logic**: Queries deals where `expiration_date` or `creditDecisionExpiryDate` has passed
- **Action**: Sets `hyper_front_status: "Expired"` + `to_be_expired: true` + `hyper_front_action: null`
- **Exclusion**: Skips deals already marked as "Expired", "Rejected", or "Cancelled"

```typescript
const expiredUpdate = {
  hyper_front_status: "Expired",
  hyper_front_action: null,
  to_be_expired: true,
  last_update_time: new Date(),
};
await db.collection("deal").updateOne(
  { miles_sales_quote_id: deal.miles_sales_quote_id },
  { $set: expiredUpdate },
);
```

### B) MongoDB Change Stream (Real-time)
- **File**: `server/plugins/mongoose.ts`
- Listens to changes on the `deal` collection via change stream
- Function `isExpiredByDateForWorkflow()` checks expiration dates per workflow
- If expired, triggers `updateDeal()` which updates the status
- **Resolution logic**:
  - `simulation` workflow → uses `deal.expiration_date`
  - `proposal` / `delivery` workflow → uses `deal.creditDecisionExpiryDate`

### Complete Data Flow

```
MongoDB (deal collection)
  ├── hyper_front_status: "Expired"  ← stored value read by frontend
  ├── expiration_date                ← original field (simulation)
  └── creditDecisionExpiryDate       ← original field (proposal/delivery)
           │
           ▼
offerStore.offerStatus         ← set verbatim from backend response
           │
           ▼
useDealStatus.isExpired        ← case-insensitive comparison (fix applied)
           │
           ▼
showModifyButton / shouldShowModifyBtn  ← Modify button hidden
```

### Unified Status Source
All three terminal statuses (Canceled/Cancelled, Rejected, Expired) originate from MongoDB's `hyper_front_status` field, which is populated by backend services and returned verbatim to the frontend via API responses.

## Post-Fix Investigation: "Cancelled" → "Canceled" Persistence Issue

### Problem
After fixing the case-insensitive status comparison in the code, the Offer Status label still displayed "Cancelled" instead of "Canceled".

### Root Cause
Two issues were identified:

#### Issue 1: Matrix CSV changed but MongoDB not re-seeded
- **File**: `scripts/data/matrix.csv` had 10 rows with `hyper_front_status: "Cancelled"` in the output column
- These were corrected to `"Canceled"` in the CSV
- But the seed script `scripts/insertMatrixData.cjs` was **never re-executed**
- The MongoDB `matrix` collection still had the old "Cancelled" values
- Since `updateDealFromMatrix()` reads from the MongoDB `matrix` collection, it kept returning "Cancelled"

**Fix**: `node scripts/insertMatrixData.cjs` → Re-populated MongoDB matrix collection from corrected CSV

#### Issue 2: Existing deals persisted with "Cancelled"
- Deals created before the fix had "Cancelled" stored directly in `deal.hyper_front_status`
- The change stream only triggers on new updates, not retroactively

**Fix**: Direct MongoDB update: `db.deal.updateMany({ hyper_front_status: "Cancelled" }, { $set: { hyper_front_status: "Canceled" } })`

### Complete Data Flow
```
BS API (Miles/Sofico)
    │  returns status enum IDs
    ▼
buildDealPayload() → saveDealToDB()
    │  hyper_front_status: null (NOT set during creation)
    ▼
MongoDB Change Stream (server/plugins/mongoose.ts)
    │  detects insert → calls updateDealFromMatrix(deal)
    ▼
updateDealFromMatrix() (server/services/dealService.ts:554-591)
    │  queries matrix collection with status IDs
    ▼
MongoDB collection "matrix"
    │  seeded from scripts/data/matrix.csv via insertMatrixData.cjs
    │  returns hyper_front_status, hyper_front_action, overview_status
    ▼
Deal document updated
    │  hyper_front_status ← "Canceled" (from matrix match)
    │  overview_status ← (from matrix match)
    │  hyper_front_action ← (from matrix match)
    ▼
Frontend (offerStore.offerStatus)
    │  receives hyper_front_status verbatim via API
    ▼
OfferPreview.vue / Deal list
    │  displays raw value as "Offer Status" label
```

### Key Insight: Async Population
The `hyper_front_status` field is **never set during deal creation** (`buildDealPayload()` does not include it). It is always populated **asynchronously** via the MongoDB change stream mechanism. This is by design.

### Matrix CSV Structure
The matrix is a mapping table with:
- **Input fields**: `sales_quote_status`, `sales_quote_status_enum_id`, `credit_application_status`, `pending_contract_status`, `credit_decision`
- **Output fields**: `hyper_front_status`, `hyper_front_action`, `overview_status`

### Possible Causes for Missing hyper_front_status
1. Deals without corresponding matrix entry (no matching status combination)
2. Deals created before matrix system existed (legacy data)
3. Change stream failure (server restart, network issue)
4. Direct database inserts bypassing the change stream
