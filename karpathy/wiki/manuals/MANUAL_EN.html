Version: 2.3 | Last Updated: 2026-05-14

§1. Overview

This workflow is an automated implementation system for Jira tickets that coordinates multiple AI agents to analyze requirements, implement features, and validate results using Playwright E2E tests.

§3. Agent Descriptions and Functions

🎯
workflow-orchestrator
Main orchestrator
deepseek-v4-flash ⏱ 10min 🔄 3 retries

Full workflow coordinator. Receives Jira tickets, delegates tasks, manages error handling and correction cycles, requests human approval when needed.

📚
wiki-keeper
Knowledge manager
qwen3.5-flash ⏱ 5min 🔄 4 retries

Maintains the workflow wiki. Creates, updates, and queries notes. Processes PDFs, emails (.eml), and documentation. Auto-syncs with Obsidian.

🔍
miles-expert
Domain specialist
minimax-m2.7 / V4 Pro ⏱ 10-15min 🔄 3 retries

Expert in MMP APIs, EU auto leasing. Auto-selects model: M2.7 (simple) or V4 Pro (complex/cross-module/high risk).

🧪
e2e-runner
E2E tester
step-3.5-flash ⏱ 15min 🔄 2 retries

Runs Playwright E2E tests against acceptance criteria. Validates full flows, takes screenshots, and reports detailed results.

  • Manage error handling and correction cycles

  • If model error: switch to fallback model

  • Orchestrator delegates to e2e-runner for testing
  • Orchestrator invokes wiki-keeper at END to create ticket notes
  • If tests fail, orchestrator loops back to miles-expert

5.1 workflow-jira-ticket (SKILL)

AttributeValue
TypeSkill (iterative workflow)
Retry2
Timeout30 minutes
PurposeImplement Jira tickets AC by AC

4 Operation Modes:

ModeDescription
bugBug fix for ticket without ACs
validarValidate existing branch (checklist + screenshots)
continuarResume incomplete implementation from history

Auto-Detect Project Type:

The workflow automatically detects the project type:

Detected TypeFile IndicatorTest FrameworkCommand
java-spring-backendpom.xml + src/test/javaJUnit/Maven./mvnw test
node-backendpackage.json (no playwright)Jest/Mochanpm test

Default paths:

  • Frontend: ~/Development/teamwill/mobilize/hyperfront
  • Backend (Java): ~/Development/teamwill/mobilize/deal-bs
  • Backend (Node): detected from project root
Full Flow:
  • mode-selection: User selects mode (nova/bug/validar/continuar)
0.1 check-project-guidelines: Read AGENTS.md for project guidelines

0.2 check-app-running: Check if app is running (validate mode only)

0.3 extract-jira-ticket: Extract ticket details, ACs, attachments

0.4 check-all-done: Check if all ACs completed

0.5 analyze-with-miles-expert (MANDATORY - nova/bug/continuar modes only, NOT validar):

- Deep domain analysis via miles-expert (ALWAYS executed)

- Input: title, description, ACs, current_ac, project_type

- Output: domain_analysis, risk_areas, dependencies

- Used to inform create-plan regardless of complexity

Validate Mode (1):
  • validate-branch: Run e2e-validator with all ACs → create wiki report → STOP
Nova/Bug Modes (2-10):
  • check-existing-plan: Verify if plan already exists
  • create-plan: Generate implementation plan for current AC
  • validate-plan: Validate plan correctness (2 retries)
  • request-human-approval: REQUIRED - Get approval before executing
- CRITICAL: NO code changes allowed before approved == true
  • execute-plan: Implement the code (ONLY AFTER approved == true)
  • e2e-validator: Run E2E tests (calls e2e-runner agent)
7.5 generate-regression-test: Auto-generate regression test after successful validation

- Copies template to playwright/tests/regression/{ticket_id}_ac{index}.spec.ts

- Includes full ticket flow and validator assertions

- Automatically run in step 8c

  • review-implementation: Review code changes (2 retries)
8b. code-quality-checker: Check code principles (DRY, KISS, YAGNI, SOLID, SoC) AND run SonarQube automatically

8c. run-regression-tests: Execute regression tests

8d. log-history: Update implementation history → next AC

KISSFunctions < 50 linesMethods < 30 lines
YAGNINo unused importsNo unused imports/methods

Reference Files:

  • Backend: deal-bs/AGENTS.md

      This section explains when it's preferable to invoke a skill in isolation versus an agent directly.

      AbstractionHigher (includes guidelines)Lower (direct prompt)

      Use skills when:

      ScenarioRecommended SkillReason
      Verify code principlesskill.code-quality-checkerAuto-detects project_type, runs SonarQube

      ScenarioAgentReason

      Agent Advantages:
      • More prompt flexibility

      Practical Example

      PARTIALLY - Agents require some form of input, but can be invoked with minimal context:

      AgentMinimum InputCan Work Without Explicit Prompt?

      §8. Error Handling and Recovery

      8.1 When an Agent Fails:

      • Check if it's a timeout or model error
      • If timeout: retry with same model (up to retry limit)
      • If model error: switch to fallback model
      • Log the error in workflow history
      • If all retries exhausted: request human intervention

      8.2 Rate Limiting

      • Each agent has request limits per minute
      • If rate limited: wait 30s before retry
      • If persistent: escalate to human

      8.3 Emergency Stop

      To cancel workflow at any time, user can type: /stop or /cancel

      • Current agent will finish gracefully
      • Progress will be logged for later resume
      • User will be notified of incomplete state

      §9. GitNexus - Code Intelligence

      GitNexus provides deep code analysis capabilities for the workflow. It creates a knowledge graph of your codebase, enabling blast radius analysis, dependency tracking, and semantic code search.

      9.1 Installation and Setup

      
# Install CLI globally
npm install -g gitnexus

# Index repositories (run from each repo root)
cd ~/Development/teamwill/mobilize/hyperfront && npx gitnexus analyze
cd ~/Development/teamwill/mobilize/deal-bs && npx gitnexus analyze

# Start local server with web UI
gitnexus serve
      Web UI: http://localhost:4747

{

      9.5 Heuristics for Suggestion

      Use GitNexus when:
      • Impact multi-module detected (affects >1 cluster)
      • Bug with non-obvious root cause
      • Refactoring structural changes
      • Low familiarity with code zone
      • Need blast radius before planning
      • High regression risk
      Don't use when:
      • Small, localized change
      • Clearly known area
      • Simple, isolated, low-risk ticket
